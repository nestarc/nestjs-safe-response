## 2. 시장 리서치 핵심 발견

### 2.1 핵심 수치

| 지표 | 수치 |
|------|------|
| @nestjs/core 주간 다운로드 | ~850만 |
| 응답 표준화 라이브러리 최대 GitHub 스타 | 11 (nest-standard-response) |
| nestjs-i18n 주간 다운로드 | 12~28만 |
| nestjs-cls 주간 다운로드 | ~40만 |
| nestjs-paginate + nestjs-typeorm-paginate 합산 | ~13.4만/주 |
| "NestJS response interceptor" 튜토리얼 (2024-2025) | 10편 이상 |

### 2.2 시장 기회

1. **시장에 승자가 없다** — 850만 사용자 vs 11스타 최대 경쟁자. 파편화된 시장.
2. **같은 튜토리얼이 반복 작성된다** — 개발자들이 매년 동일한 인터셉터를 처음부터 구현. 패키지가 필요하다는 가장 강한 신호.
3. **NestJS 팀이 명시적으로 커뮤니티에 위임** — nestjs/nest #15244에서 "응답 처리는 인터셉터/라이브러리 영역"이라고 공식 답변. 코어에 들어갈 일 없음.
4. **Swagger + 제네릭 래퍼 = 7년간 미해결** — nestjs/swagger #86 (2018년~), 42+ 관련 이슈. 우리는 이미 해결함.

### 2.3 경쟁사가 못 하는 것 (v0.8.0 기준)

| 영역 | nestjs-safe-response | 경쟁사 |
|------|---------------------|--------|
| Swagger + 제네릭 래퍼 | 완전 자동 | 모두 수동 또는 미지원 |
| 성공 래핑 + 에러 포맷 통합 | 단일 모듈 | 분리되거나 한쪽만 |
| RFC 9457 | 성공 래핑과 통합 | nest-problem-details는 에러만 |
| ORM 무관 페이지네이션 | 오프셋 + 커서 | nestjs-paginate는 TypeORM 전용 |
| 듀얼 어댑터 | Express + Fastify | 대부분 Express 전용 |

## 3. v0.9.0 제안: 4개 테마

### 테마 A: 글로벌 에러 응답 자동 문서화 (Swagger DX)

**영향도: 높음 | 난이도: 중 | 수요 근거: nestjs/swagger #870, #1484**

#### 문제

현재 모든 라우트에 공통 에러(401, 403, 500)를 수동으로 추가해야 한다:

```typescript
// 현재: 모든 컨트롤러마다 반복
@ApiSafeErrorResponses([401, 403, 500])
@Get()
findAll() { ... }

@ApiSafeErrorResponses([401, 403, 500])
@Post()
create() { ... }
```

#### 해결

모듈 등록 시 글로벌 에러 응답을 한 번만 선언:

```typescript
SafeResponseModule.register({
  swagger: {
    globalErrors: [401, 403, 500],
    // 또는 상세 설정:
    globalErrors: [
      401,
      { status: 403, code: 'FORBIDDEN', description: 'Insufficient permissions' },
      { status: 500, description: 'Unexpected server error' },
    ],
  },
})
```

#### 구현 항목

| 항목 | 설명 |
|------|------|
| `SwaggerOptions` 인터페이스 | `globalErrors` 배열 옵션 |
| Swagger 플러그인 (DocumentBuilder 확장) | `SwaggerModule.createDocument()` 후 스키마 주입 |
| `@SkipGlobalErrors()` 데코레이터 | 특정 라우트에서 글로벌 에러 제외 (예: 헬스체크) |
| 기존 `@ApiSafeErrorResponse`와 병합 | 라우트별 에러 + 글로벌 에러 합산 |

#### API 설계

```typescript
// 1. 모듈 옵션
interface SwaggerOptions {
  /** Error responses to add to all routes */
  globalErrors?: ApiSafeErrorResponseConfig[];
}

interface SafeResponseModuleOptions {
  // ... 기존 옵션
  /** Swagger documentation options */
  swagger?: SwaggerOptions;
}

// 2. Swagger 문서 생성 시 적용
import { SafeResponseModule } from 'nestjs-safe-response';

const document = SwaggerModule.createDocument(app, config);
SafeResponseModule.applyGlobalErrors(document); // 글로벌 에러 스키마 주입

// 3. 특정 라우트 제외
@SkipGlobalErrors()
@Get('health')
healthCheck() { return { status: 'ok' }; }
```

### 테마 B: 프론트엔드 클라이언트 타입 (DX)

**영향도: 높음 | 난이도: 하 | 수요 근거: OpenAPI → 타입 파이프라인 요구 다수**

#### 문제

프론트엔드에서 API 응답 타입을 사용하려면 `@nestjs/common`, `@nestjs/swagger` 등
백엔드 의존성을 설치해야 한다. 또는 타입을 수동으로 복사해야 한다.

#### 해결

런타임 의존성 없는 순수 TypeScript 타입을 별도 엔트리포인트로 제공:

```typescript
// 프론트엔드에서 사용
import type {
  SafeSuccessResponse,
  SafeErrorResponse,
  PaginationMeta,
  CursorPaginationMeta,
} from 'nestjs-safe-response/client';

// API 호출 래퍼
async function fetchApi<T>(url: string): Promise<SafeSuccessResponse<T>> {
  const res = await fetch(url);
  if (!res.ok) {
    const error: SafeErrorResponse = await res.json();
    throw new ApiError(error);
  }
  return res.json();
}

// 사용
const { data, meta } = await fetchApi<User[]>('/api/users');
if (meta?.pagination) {
  console.log(`Page ${meta.pagination.page} of ${meta.pagination.totalPages}`);
}
```

#### 구현 항목

| 항목 | 설명 |
|------|------|
| `src/client/index.ts` | 순수 타입 재선언 (NestJS 의존성 없음) |
| `package.json` exports 맵 | `"./client"` 엔트리포인트 추가 |
| `tsconfig.client.json` | 클라이언트 전용 빌드 설정 |
| 타입 가드 유틸리티 | `isSuccess()`, `isError()`, `isPaginated()` 함수 |

#### API 설계

```typescript
// src/client/index.ts — 순수 타입 (런타임 코드 없음)

// 응답 타입
export interface SafeSuccessResponse<T = unknown> {
  success: true;
  statusCode: number;
  code?: string;
  requestId?: string;
  data: T;
  meta?: ResponseMeta;
  timestamp?: string;
  path?: string;
}

export interface SafeErrorResponse {
  success: false;
  statusCode: number;
  requestId?: string;
  error: { code: string; message: string; details?: unknown };
  meta?: { responseTime?: number };
  timestamp?: string;
  path?: string;
}

export interface SafeProblemDetailsResponse {
  type: string;
  title: string;
  status: number;
  detail: string;
  instance: string;
  code?: string;
  requestId?: string;
  details?: unknown;
  meta?: { responseTime?: number };
}

// 페이지네이션 타입
export interface PaginationMeta { ... }
export interface CursorPaginationMeta { ... }
export interface PaginationLinks { ... }
export interface ResponseMeta { ... }

// 유니온 타입
export type SafeResponse<T = unknown> = SafeSuccessResponse<T> | SafeErrorResponse;

// 타입 가드 (런타임 유틸리티 — 이것만 실제 JS 코드)
export function isSuccess<T>(res: SafeResponse<T>): res is SafeSuccessResponse<T> {
  return res.success === true;
}

export function isError(res: SafeResponse): res is SafeErrorResponse {
  return res.success === false;
}

export function isPaginated(meta?: ResponseMeta): boolean {
  return meta?.pagination !== undefined;
}

export function isOffsetPagination(
  pagination: PaginationMeta | CursorPaginationMeta,
): pagination is PaginationMeta {
  return !('type' in pagination) || pagination.type === 'offset';
}

export function isCursorPagination(
  pagination: PaginationMeta | CursorPaginationMeta,
): pagination is CursorPaginationMeta {
  return 'type' in pagination && pagination.type === 'cursor';
}
```

#### package.json exports 설정

```jsonc
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "default": "./dist/index.js"
    },
    "./client": {
      "types": "./dist/client/index.d.ts",
      "default": "./dist/client/index.js"
    }
  }
}
```

### 테마 C: nestjs-i18n 에러 메시지 연동

**영향도: 높음 | 난이도: 중 | 수요 근거: nestjs-i18n 12~28만 주간 DL**

#### 문제

프로덕션 앱에서 에러 메시지를 사용자 언어로 표시해야 한다.
현재는 영어 고정이거나 수동으로 번역 로직을 삽입해야 한다.

#### 해결

`nestjs-i18n`이 설치되어 있으면 에러 메시지를 자동 번역하는 옵트인 옵션:

```typescript
SafeResponseModule.register({
  i18n: true, // nestjs-i18n이 설치되어 있으면 자동 번역 활성화
})
```

#### 구현 전략: 어댑터 패턴

`nestjs-i18n`을 **peerDependency (optional)**로 두고, 어댑터 패턴으로 결합도를 최소화:

```typescript
// 인터페이스 정의 (nestjs-i18n 의존 없음)
export interface I18nAdapter {
  translate(key: string, options?: { lang?: string; args?: Record<string, unknown> }): string;
  resolveLanguage(request: unknown): string;
}

// 모듈 옵션
interface SafeResponseModuleOptions {
  // ... 기존 옵션
  /** Enable i18n for error messages. Pass true to auto-detect nestjs-i18n, or provide a custom adapter. */
  i18n?: boolean | I18nAdapter;
}
```

#### 사용 예시

```typescript
// 방법 1: 자동 감지 (nestjs-i18n이 DI 컨테이너에 있으면 자동 사용)
SafeResponseModule.register({
  i18n: true,
})

// 방법 2: 커스텀 어댑터 (다른 i18n 라이브러리 사용 시)
SafeResponseModule.register({
  i18n: {
    translate: (key, opts) => myI18nService.t(key, opts?.lang),
    resolveLanguage: (req) => req.headers['accept-language'] ?? 'en',
  },
})
```

#### 에러 메시지 번역 흐름

```
1. 예외 발생: throw new BadRequestException('errors.VALIDATION_FAILED')
2. SafeExceptionFilter가 캐치
3. i18n 어댑터 존재 여부 확인
4. 존재하면: 요청의 Accept-Language → 해당 언어로 번역
5. 미존재하면: 원본 메시지 그대로 사용 (현재 동작)
```

#### 번역 파일 예시 (사용자 프로젝트)

```json
// i18n/ko/errors.json
{
  "VALIDATION_FAILED": "입력값 검증에 실패했습니다",
  "NOT_FOUND": "요청한 리소스를 찾을 수 없습니다",
  "UNAUTHORIZED": "인증이 필요합니다"
}
```

#### 응답 예시

```jsonc
// Accept-Language: ko
{
  "success": false,
  "statusCode": 400,
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "입력값 검증에 실패했습니다"  // 번역됨
  }
}

// Accept-Language: en (또는 i18n 미설정)
{
  "success": false,
  "statusCode": 400,
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Validation failed"  // 원본
  }
}
```

#### 구현 항목

| 항목 | 설명 |
|------|------|
| `I18nAdapter` 인터페이스 | 번역 + 언어 감지 계약 |
| `NestI18nAdapter` 내부 구현 | `nestjs-i18n`의 `I18nService` 래퍼 |
| `SafeExceptionFilter` 확장 | `catch()` 내에서 어댑터로 메시지 번역 |
| 자동 감지 로직 | `ModuleRef.get(I18nService, { strict: false })` |
| RFC 9457 `detail` 필드도 번역 | Problem Details 모드에서도 동일 적용 |
| 설정 문서 + 예제 | README에 i18n 섹션 추가 |

### 테마 D: nestjs-cls 컨텍스트 연동

**영향도: 중~높음 | 난이도: 하 | 수요 근거: nestjs-cls 40만 주간 DL**

#### 문제

마이크로서비스/분산 시스템에서 `traceId`, `correlationId`, `userId` 등
요청 컨텍스트를 응답에 자동 포함해야 한다.
현재 `requestId`는 자체 구현되어 있지만, OpenTelemetry traceId 같은
외부 컨텍스트와 통합되지 않는다.

#### 해결

`nestjs-cls`의 `ClsService`에서 값을 읽어 응답 `meta`에 자동 주입:

```typescript
SafeResponseModule.register({
  context: {
    // CLS 스토어에서 읽을 키 → 응답 meta에 매핑
    fields: {
      traceId: 'traceId',           // cls.get('traceId') → meta.traceId
      correlationId: 'correlationId', // cls.get('correlationId') → meta.correlationId
    },
    // 또는 커스텀 리졸버
    resolver: (clsService) => ({
      traceId: clsService.get('traceId'),
      userId: clsService.get('userId'),
    }),
  },
})
```

#### 구현 전략

`nestjs-cls`를 **optional peerDependency**로 두고, 인터셉터에서 CLS 값 읽기:

```typescript
// 인터페이스
export interface ContextOptions {
  /** Map CLS store keys to response meta fields */
  fields?: Record<string, string>;
  /** Custom resolver function (receives ClsService or similar) */
  resolver?: (store: unknown) => Record<string, unknown>;
}

interface SafeResponseModuleOptions {
  // ... 기존 옵션
  /** Inject request context values into response meta */
  context?: ContextOptions;
}
```

#### 응답 예시

```jsonc
{
  "success": true,
  "statusCode": 200,
  "requestId": "abc-123",
  "data": { "..." },
  "meta": {
    "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",  // OpenTelemetry에서 주입
    "correlationId": "order-flow-789",                 // 비즈니스 컨텍스트
    "responseTime": 42
  }
}
```

#### 구현 항목

| 항목 | 설명 |
|------|------|
| `ContextOptions` 인터페이스 | 필드 매핑 + 커스텀 리졸버 |
| `SafeResponseInterceptor` 확장 | `map()` 내에서 CLS 값 읽기 |
| `SafeExceptionFilter` 확장 | 에러 응답에도 컨텍스트 포함 |
| `ResponseMeta` 인터페이스 확장 | `[key: string]: unknown` 인덱스 시그니처 |
| `ResponseMetaDto` 확장 | Swagger에 커스텀 필드 반영 (additionalProperties) |
| 자동 감지 로직 | `ModuleRef.get(ClsService, { strict: false })` |

## 4. 추가 포함 사항

### 테마 E: Sort/Filter 메타데이터

```typescript
@Get()
@Paginated({ maxLimit: 100 })
@SortMeta()      // 현재 적용된 정렬 정보를 meta에 포함
@FilterMeta()    // 현재 적용된 필터 정보를 meta에 포함
@ApiPaginatedSafeResponse(UserDto)
async findAll(
  @Query('sort') sort = 'createdAt:desc',
  @Query('filter') filter?: string,
) {
  return { data: users, total, page, limit, sort, filters };
}
```

응답 예시:
```jsonc
{
  "meta": {
    "pagination": { "..." },
    "sort": { "field": "createdAt", "order": "desc" },
    "filters": { "status": "active", "role": "admin" }
  }
}
```

| 항목 | 설명 |
|------|------|
| `@SortMeta()` 데코레이터 | 정렬 정보를 meta에 포함 |
| `@FilterMeta()` 데코레이터 | 필터 정보를 meta에 포함 |
| `SortMetaDto`, `FilterMetaDto` | Swagger 스키마 자동 생성 |
| `PaginatedResult` 확장 | `sort?`, `filters?` 필드 추가 |

|------|------------|
| 글로벌 에러 문서화 | `swagger.globalErrors` 미설정 시 현재 동작 유지 |
| 클라이언트 타입 | 새 엔트리포인트 `./client` 추가 — 기존 import 영향 없음 |
| i18n 연동 | `i18n: false`(기본값) — 옵트인 방식 |
| CLS 연동 | `context` 미설정 시 현재 동작 유지 |
| `ResponseMeta` 확장 | 인덱스 시그니처 추가는 기존 타입에 호환 |

## 6. 구현 순서

의존 관계와 난이도를 고려한 구현 순서:

```
Phase 1: 프론트엔드 클라이언트 타입 (테마 B)
  └─ 가장 독립적, 난이도 낮음, 즉시 가치 제공
  └─ package.json exports 변경은 다른 작업에 영향 없음

Phase 2: 글로벌 에러 응답 문서화 (테마 A)
  └─ Swagger 영역만 변경, 런타임 영향 없음
  └─ 테마 B의 클라이언트 타입에 에러 타입이 이미 포함됨

Phase 3: nestjs-cls 컨텍스트 연동 (테마 D)
  └─ 인터셉터/필터 확장, ResponseMeta 인터페이스 변경
  └─ i18n보다 단순 (번역 로직 없이 값 읽기만)

Phase 4: nestjs-i18n 에러 메시지 연동 (테마 C)
  └─ 필터 확장, 가장 복잡한 외부 의존성 통합
  └─ Phase 3의 어댑터 패턴 경험을 활용

Phase 5: Sort/Filter 메타데이터 (테마 E)
  └─ Phase 3의 ResponseMeta 확장 위에 구축
  └─ @Paginated()와 자연스럽게 결합
```

## 10. Optional Peer Dependencies 전략

v0.9.0에서 외부 라이브러리 연동 시 **optional peerDependencies** 패턴을 사용:

```jsonc
// package.json
{
  "peerDependencies": {
    "@nestjs/common": "^10.0.0 || ^11.0.0",
    "@nestjs/core": "^10.0.0 || ^11.0.0",
    "@nestjs/swagger": "^8.0.0 || ^11.0.0",
    "reflect-metadata": "^0.2.0",
    "rxjs": "^7.0.0"
  },
  "peerDependenciesMeta": {
    "nestjs-i18n": { "optional": true },
    "nestjs-cls": { "optional": true }
  },
  "optionalPeerDependencies": {
    "nestjs-i18n": "^10.0.0 || ^11.0.0",
    "nestjs-cls": "^4.0.0 || ^5.0.0"
  }
}
```

**핵심 원칙**:
- 선택적 라이브러리가 없어도 **모든 기존 기능이 정상 동작**
- 런타임에 `try { require('nestjs-i18n') }` 대신 NestJS DI의 `ModuleRef.get(Service, { strict: false })` 사용
- 타입은 조건부 import로 빌드 시 에러 방지
- **버전 범위 최대화**: nestjs-i18n v10/v11, nestjs-cls v4/v5 모두 지원 (사용자 호환성 우선)

## 12. 결정 사항 (확정 — 2026-03-31)

모든 결정은 **사용자 중심** 원칙으로 확정함.
판단 기준: "이 기능을 처음 쓰는 NestJS 개발자가 가장 직관적으로 느낄 선택"

### 1. 테마 우선순위 → 기획서 순서 유지 (B → A → D → C)

- **Phase 1 (클라이언트 타입)**: 프론트엔드 개발자가 설치 즉시 `import type`으로 가치를 느낌
- **Phase 2 (글로벌 에러)**: 백엔드 개발자가 모든 컨트롤러의 `@ApiSafeErrorResponses` 반복 제거
- **Phase 3 (CLS)**: 설정 한 줄(`context: { fields: { traceId: 'traceId' } }`)로 완료
- **Phase 4 (i18n)**: 번역 파일 준비가 필요해 사용자 진입 비용이 가장 높음 → 마지막

### 2. i18n 범위 → 에러 메시지 + `meta.message` + RFC 9457 `detail` 모두 번역

사용자 관점에서 "i18n을 켰는데 에러만 번역되고 성공 메시지는 영어"는 혼란스럽다.

번역 대상:
- `error.message` — 에러 응답 메시지
- `meta.message` — `@ResponseMessage()`로 설정한 성공 메시지
- RFC 9457 `detail` — Problem Details 모드의 설명

번역하지 않는 것:
- `error.code` — 머신 리더블 코드 (`BAD_REQUEST`)는 원본 유지
- `error.details` — 검증 에러 배열은 구조화된 데이터이므로 원본 유지

### 3. CLS 컨텍스트 주입 위치 → `meta`에 주입

- 최상위 필드(`success`, `statusCode`, `data`, `requestId`)는 라이브러리가 보장하는 고정 스키마
- `meta`는 가변적 메타데이터 컨테이너 — 사용자가 추가하는 값은 여기에 속함
- `requestId`만 예외적으로 최상위인 이유: HTTP 요청 추적의 표준 관행이며 v0.7.0부터 확립된 API

### 4. Nice-to-Have → Sort/Filter(N-1) 포함, ESM(N-2) 이월

- **Sort/Filter 포함**: `@Paginated()`를 쓰는 사용자는 자연스럽게 정렬/필터 메타도 원함
- **ESM 이월**: NestJS 자체가 CJS 우선, 번들러 사용자는 이미 CJS 소비 → 사용자 체감 개선 거의 없음

### 5. Optional Peer Dependency 버전 → 호환성 최대화

- **nestjs-i18n**: `"^10.0.0 || ^11.0.0"` — v10 사용자도 연동 가능
- **nestjs-cls**: `"^4.0.0 || ^5.0.0"` — v4 다수 사용자 포함
