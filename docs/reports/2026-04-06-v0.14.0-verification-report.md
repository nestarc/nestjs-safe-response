## Findings

### 1. Newly advertised metadata fields are not consistently present in exported TypeScript interfaces

Severity: High

Files:

- `CHANGELOG.md`
- `src/interfaces/index.ts`
- `src/client/index.ts`

Evidence:

- `CHANGELOG.md` explicitly claims `ResponseMeta.apiVersion` / `ResponseMeta.fields` were added to both server and client types, and that `apiVersion` is included in error metadata for `0.14.0`.
- In `src/interfaces/index.ts`, server-side `ResponseMeta` still omits explicit `apiVersion` and `fields` members.
- In `src/interfaces/index.ts`, server-side `SafeErrorResponse.meta` and `SafeProblemDetailsResponse.meta` omit explicit `apiVersion`.
- In `src/client/index.ts`, success `ResponseMeta` includes `apiVersion` and `fields`, but client-side `SafeErrorResponse.meta` and `SafeProblemDetailsResponse.meta` still omit explicit `apiVersion`.
- Runtime behavior and Swagger DTOs already expose these members, so the typings surface now lags behind the actual response contract.

Impact:

- Consumers of the published TypeScript API lose autocomplete and strong typing for a headline `0.14.0` capability.
- Error and Problem Details consumers cannot rely on `meta.apiVersion` without falling back to the index signature.
- Because these types remain structurally assignable via `[key: string]: unknown`, the current `tsd` suite does not catch the regression.

Recommended fix:

- Add explicit `apiVersion?: string` and `fields?: string[]` members wherever the runtime/documented contract emits them.
- Extend `index.test-d.ts` with regression assertions for success, error, and Problem Details metadata.

### 2. `README.ko.md` test inventory still reflects pre-0.14.0 totals
