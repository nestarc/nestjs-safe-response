## Findings

### 1. Problem Details Swagger examples are wrong for non-404 responses

Severity: High

Files:

- `src/decorators/index.ts`
- `src/swagger/global-errors.ts`
- `src/dto/response.dto.ts`

Evidence:

- [`src/decorators/index.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/src/decorators/index.ts#L337) uses a plain `$ref` to `ProblemDetailsDto` whenever `@ApiSafeProblemResponse(status)` is called without explicit `code`/`message`/`details`.
- [`src/swagger/global-errors.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/src/swagger/global-errors.ts#L117) also uses a plain `$ref` to `ProblemDetailsDto` for `applyGlobalErrors(..., { problemDetails: true })`.
- [`src/dto/response.dto.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/src/dto/response.dto.ts#L220) hard-codes Problem Details examples as `404`, `Not Found`, `NOT_FOUND`.

Impact:

- `@ApiSafeProblemResponse(400)` documents a 400 response using 404/NOT_FOUND examples.
- Composite decorators with `problemDetails: true` inherit the same problem for numeric error configs.
- `applyGlobalErrors()` in Problem Details mode documents 401/500/etc. with 404-oriented examples.
- This does not break runtime behavior, but it weakens one of the library's core value propositions: accurate Swagger contracts.

Why this matters:

- The current Swagger tests only assert content type / `$ref`, not status-specific examples:
  - [`src/decorators/decorators.spec.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/src/decorators/decorators.spec.ts#L456)
  - [`test/e2e/swagger-schema.e2e-spec.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/test/e2e/swagger-schema.e2e-spec.ts#L280)
  - [`src/swagger/global-errors.spec.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/src/swagger/global-errors.spec.ts#L310)

Recommended fix:

- Always wrap `ProblemDetailsDto` in `allOf` with at least a status-specific override block.
- For non-404 statuses, also override `title`, `code`, and a neutral `detail` example or a per-status default.

### 2. Error/Problem `meta` Swagger DTO still does not expose arbitrary context fields

Severity: Medium

Files:

- `src/dto/response.dto.ts`
- `README.md`
- `CHANGELOG.md`

Evidence:

- [`src/dto/response.dto.ts`](/Users/ksy/Documents/GitHub/nestjs-safe-response/src/dto/response.dto.ts#L186) defines `ErrorResponseMetaDto` with fixed known properties only.
- The docs promise arbitrary CLS-derived context values in `meta`:
  - [`README.md`](/Users/ksy/Documents/GitHub/nestjs-safe-response/README.md#L624)
  - [`CHANGELOG.md`](/Users/ksy/Documents/GitHub/nestjs-safe-response/CHANGELOG.md#L123)

Impact:

- Runtime error responses can include context fields such as `traceId`, but Swagger for `SafeErrorResponseDto` and `ProblemDetailsDto` does not declare those extra properties.
- Generated clients or schema consumers may treat those fields as undocumented / disallowed on error responses.

Recommended fix:

- Add `additionalProperties: true` support to `ErrorResponseMetaDto`, matching the runtime type surface and existing `ProblemDetailsDto` fallback schema in `applyGlobalErrors()`.
