# nestjs-safe-response v0.5.0+ 리서치 리포트

## 현재 위치 (v0.5.0 완료)

v0.1.0~v0.5.0까지 **핵심 기반**이 완성되었다:

- 응답 래핑 + 에러 표준화 + 페이지네이션 + Swagger 스키마 자동 생성
- Express/Fastify 양쪽 지원, RPC/WebSocket 안전 분기
- NestJS v10/v11 × Swagger v8/v11 호환 검증 완료
- `transformResponse`, `successCodeMapper`, `@SuccessCode()` 등 DX 기능

로드맵상 다음은 **v1.0.0 (API 안정화 + 정식 릴리스)**.

## NestJS 생태계 주요 불편 사항

| 순위 | 불편 사항 | 심각도 | 현재 해결 여부 |
|------|-----------|--------|---------------|
| 1 | **Swagger에서 Generic 타입 미지원** — `ApiResponse<T>` 같은 래퍼를 쓸 수 없어 모든 응답마다 별도 DTO 필요 | 매우 높음 | **이미 해결** (`@ApiSafeResponse(Dto)`로 자동 생성) |
| 2 | **표준화된 에러 응답 없음** — 프로젝트마다 에러 포맷 재발명 | 높음 | **이미 해결** (에러 코드 매핑 + validation details) |
| 3 | **페이지네이션 표준 부재** + Swagger 문서화 이중고 | 높음 | **이미 해결** (`@Paginated()` + `@ApiPaginatedSafeResponse()`) |
| 4 | **에러 응답 Swagger 문서화 불가** — `@ApiException()` 같은 데코레이터가 없음 (nestjs/swagger Issue #870, 2020년부터 미해결) | 중간 | **미해결** |
| 5 | **프론트엔드 타입 연동 파이프라인 부재** — Swagger JSON → 타입 생성 자동화 | 중간 | **미해결** |
| 6 | **class-transformer 중첩 객체 문제** — `@Type()` 누락 시 사일런트 실패 | 중간 | 범위 밖 |
| 7 | **커서 기반 페이지네이션** — offset 방식의 한계 (데이터 변동 시 중복/누락) | 중간 | **미해결** |

### 세부 조사

### 1. 응답 처리 및 직렬화

- `ClassSerializerInterceptor`가 래핑된 응답(`{ data, meta }`)과 충돌
- 중첩 DTO에 `@Type()` 데코레이터 누락 시 plain object로 사일런트 실패
- 모노레포에서 class-transformer 메타데이터 격리 문제
- `@Exclude()` / `@Expose()` 동작이 `excludeExtraneousValues` 설정에 따라 혼란

### 2. DTO 검증 및 변환 ("데코레이터 지옥")

- **타입 정보 3중 선언**: TypeScript 타입 + `@IsNumber()` (검증) + `@Type(() => Number)` (변환)
- 자동 추론 플러그인 요청(NestJS Issue #15329)이 미구현 상태
- `PickType`, `OmitType`, `PartialType` 사용 시 `@Transform` 데코레이터 상속 깨짐
- Zod/TypeBox 등 스키마 기반 검증으로의 전환 추세 존재

### 3. Swagger/OpenAPI 문서 정확성

- **Generic 타입 미지원** (nestjs/swagger Issue #86, 2018년부터 미해결) — 가장 큰 구조적 한계
- `@nestjs/swagger/plugin`과 수동 `@ApiOkResponse()` 병용 시 타입 정보 덮어쓰기
- 글로벌 응답 envelope 정의 방법 없음 (Issue #1484, #2548)
- TypeScript 타입 별칭은 데코레이터에 사용 불가 (런타임 값인 클래스만 가능)

### 4. 프론트엔드 타입 안전성

- OpenAPI → TypeScript 파이프라인(데코레이터 → Swagger JSON → 코드 생성기 → 프론트 타입)의 취약성
- Swagger 모듈이 spec 파일 자동 생성 미지원 (라이브 UI만 제공)
- CI 강제 없으면 스펙과 실제 응답 간 드리프트 발생
- 대안: tRPC(모노레포 필수), ts-rest(추상화 레이어 추가), oRPC(미성숙)

### 5. 에러 처리 패턴

- `app.useGlobalFilters()`로 등록 시 DI 접근 불가 (`APP_FILTER` 프로바이더 패턴 필요)
- NestJS 기본 에러 포맷(`{ statusCode, message, error }`)이 실무에 부족
- ValidationPipe 거부 시 구조화된 필드별 에러 정보 얻기 어려움
- `@ApiException()` 데코레이터 부재 (nestjs/swagger Issue #870)

### 6. 페이지네이션

- NestJS 기본 페이지네이션 지원 없음 — 프로젝트마다 재구현
- offset 페이지네이션은 데이터 변동 시 중복/누락 발생
- `nestjs-paginate`는 TypeORM 전용, 커서 컬럼 타입 제한
- 페이지네이션 + Swagger 문서화 조합이 특히 고통스러움

### 7. 기존 라이브러리 비교

| 라이브러리 | 기능 | 한계 |
|-----------|------|------|
| nest-standard-response | 메타데이터 기반 래핑 + 페이지네이션/정렬/필터 | Generic 미지원, 수동 DTO 업데이트 필요 |
| @zabih-dev/nest-api-response | 단순 인터셉터 래핑 | 페이지네이션 없음, Swagger 지원 제한 |
| nestjs-swagger-dto | swagger + validator + transformer 데코레이터 통합 | 여전히 클래스 기반 |
| nestjs-paginate | TypeORM 페이지네이션 + 필터링 | TypeORM 전용, 커서 컬럼 타입 제한 |
| nestjs-trpc | NestJS에 tRPC 통합 | 모노레포 필수, OpenAPI 스펙 없음 |
| ts-rest | Zod 기반 계약 우선 타입 안전 REST | 추상화 레이어 추가, NestJS 네이티브 아님 |

## v0.5.0 → v1.0.0 기능 후보

### A. 높은 임팩트 + 라이브러리 방향과 일치

#### 1. `@ApiSafeErrorResponse()` 데코레이터 ⭐ 최우선 추천

```typescript
@ApiSafeErrorResponse(400, { code: 'VALIDATION_ERROR', description: '입력값 검증 실패' })
@ApiSafeErrorResponse(404, { code: 'NOT_FOUND', description: '리소스 없음' })
```

- nestjs/swagger Issue #870 — 오래된 커뮤니티 요청
- 성공 응답 Swagger는 해결했지만 **에러 응답 Swagger는 미해결** (비대칭)
- 기존 `errorCodeMapper`와 자연스럽게 연동 가능
- 구현 난이도 대비 사용자 체감 개선이 큼

#### 2. 커서 기반 페이지네이션 지원

```typescript
@Paginated({ strategy: 'cursor' })  // vs 기존 'offset'
```

- 기존 `@Paginated()`는 offset 기반만 지원
- 무한 스크롤, 실시간 데이터에서 offset 한계 명확
- `nestjs-paginate`는 TypeORM 종속 — ORM 무관 추상화가 차별점

#### 3. OpenAPI spec 자동 내보내기 유틸리티

```bash
npx nestjs-safe-response generate-spec  # → openapi.json 생성
```

- 프론트엔드 타입 생성 파이프라인의 시작점
- Orval / openapi-typescript와 연동 가이드

### B. 중간 임팩트 + DX 개선

#### 4. 응답 헤더 자동 설정

- `X-Request-Id`, `X-Total-Count` 등 표준 헤더 자동 추가
- 페이지네이션 시 `Link` 헤더 (RFC 5988) 자동 생성

#### 5. 응답 버저닝 지원

```typescript
@ApiSafeResponse(UserDtoV2, { version: '2' })
```

- API 버전별 다른 응답 스키마 문서화

#### 6. 타입 안전한 클라이언트 헬퍼

```typescript
// 프론트엔드에서 사용 — 런타임 의존성 없이 타입만 제공
import type { SafeResponse, PaginatedResponse } from 'nestjs-safe-response/client';
const res: SafeResponse<User> = await fetch('/api/users/1').then(r => r.json());
```

- 별도 `client` 엔트리포인트

### C. v1.0.0 안정화 필수 항목

#### 7. API 표면 동결 + Migration 가이드

- 모든 public API의 JSDoc 완성
- v0.x → v1.0 마이그레이션 문서

#### 8. 벤치마크

- Interceptor/Filter의 오버헤드 측정
- "래핑 비용이 무시할 만하다"는 근거 데이터 제공
