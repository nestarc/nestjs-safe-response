# v0.13.0 Design Spec: DX Improvements

**Date:** 2026-04-04
**Version:** 0.13.0
**Theme:** Developer Experience improvements — silent failure prevention, composite decorators, error code mapping

## Overview

v0.13.0 focuses on three DX improvements that address the most impactful pain points identified through ecosystem research and codebase analysis. All changes are **non-breaking** and purely additive.

## 1. Silent Failure Prevention

### Problem

When `@Paginated()` is applied to a handler but the handler returns data that does not match `PaginatedResult` shape, the interceptor silently skips pagination metadata. No error, no warning. The developer sees a normal success response without pagination meta and has no indication of what went wrong. Same issue affects `@CursorPaginated()`, `@SortMeta()`, and `@FilterMeta()`.

### Solution

Add `Logger.warn()` calls in the interceptor's `map()` callback when decorator metadata is present but data shape validation fails. Include the request method + path and the expected shape in the warning message.

### Affected Decorators

| Decorator | Validation | Warning Trigger |
|-----------|-----------|----------------|
| `@Paginated()` | `isPaginatedResult(data)` fails | Data lacks `data: T[]`, `total: number`, `page: number`, or `limit: number` |
| `@CursorPaginated()` | `isCursorPaginatedResult(data)` fails | Data lacks `data: T[]`, `nextCursor: string\|null`, `hasMore: boolean`, or `limit: number` |
| `@SortMeta()` | `data.sort` missing | Data object does not have a `sort` field |
| `@FilterMeta()` | `data.filters` missing | Data object does not have a `filters` field |

### Warning Format

```
[SafeResponseInterceptor] @Paginated() on GET /api/users: handler returned data that does not match PaginatedResult shape. Expected { data: T[], total: number, page: number, limit: number }. Pagination metadata will be skipped.
```

### Opt-out

```typescript
SafeResponseModule.register({
  suppressWarnings: true,  // default: false
})
```

`suppressWarnings` is a new optional boolean field on `SafeResponseModuleOptions`. When `true`, all shape-mismatch warnings are suppressed.

### Implementation Details

- Logger instance: `private readonly logger = new Logger(SafeResponseInterceptor.name);` (already exists on filter, add to interceptor)
- Warning emitted on every request with a mismatch (not just the first). Rationale: intermittent issues (e.g., sometimes the service returns the correct shape, sometimes not) should be visible every time.
- Warning runs after `transformResponse` hook (data may have been transformed before shape check).
- `null` and `undefined` data trigger the warning when pagination decorators are present (these are clearly not paginated results).

### Files Modified

- `src/interceptors/safe-response.interceptor.ts` — add Logger, add 4 warning checks in `map()` callback
- `src/interfaces/index.ts` — add `suppressWarnings?: boolean` to `SafeResponseModuleOptions`

### Tests

- 4 unit tests: one per decorator, verifying `Logger.warn()` is called with the correct message
- 4 unit tests: verifying warnings are suppressed when `suppressWarnings: true`
- 1 unit test: verifying no warning when data matches expected shape
- 1 E2E test: handler with `@Paginated()` returning non-paginated data produces warning in logs

## 2. Composite Decorators

### Problem

A paginated endpoint requires stacking 4-5+ decorators: `@ApiPaginatedSafeResponse(Dto)` + `@Paginated()` + `@SortMeta()` + `@FilterMeta()` + `@ApiSafeErrorResponses([...])`. This is verbose and error-prone (forgetting one is a common mistake).

### Solution

Three composite decorators that combine runtime behavior + Swagger documentation in a single annotation. Existing individual decorators remain fully supported.

### 2.1 `@SafeEndpoint(Model, opts?)`

For standard (non-paginated) endpoints.

```typescript
export function SafeEndpoint<T extends Type>(
  model: T,
  options?: SafeEndpointOptions,
): MethodDecorator;
```

**Options interface:**

```typescript
export interface SafeEndpointOptions {
  /** HTTP status code for Swagger response (default: 200) */
  statusCode?: number;
  /** Whether data is an array (default: false) */
  isArray?: boolean;
  /** Swagger response description */
  description?: string;
  /** Include sort metadata from handler return value (default: false) */
  sort?: boolean;
  /** Include filter metadata from handler return value (default: false) */
  filter?: boolean;
  /** Custom response message in meta */
  message?: string;
  /** Custom success code */
  code?: string;
  /** Error responses to document in Swagger */
  errors?: ApiSafeErrorResponseConfig[];
  /** Mark endpoint as deprecated with RFC headers */
  deprecated?: DeprecatedOptions;
}
```

**Internal composition:**

```typescript
export function SafeEndpoint<T extends Type>(
  model: T,
  options: SafeEndpointOptions = {},
): MethodDecorator {
  const decorators: MethodDecorator[] = [
    ApiSafeResponse(model, {
      isArray: options.isArray,
      statusCode: options.statusCode,
      description: options.description,
    }),
  ];

  if (options.sort) decorators.push(SortMeta());
  if (options.filter) decorators.push(FilterMeta());
  if (options.message) decorators.push(ResponseMessage(options.message));
  if (options.code) decorators.push(SuccessCode(options.code));
  if (options.errors?.length) decorators.push(ApiSafeErrorResponses(options.errors));
  if (options.deprecated) decorators.push(Deprecated(options.deprecated));

  return applyDecorators(...decorators);
}
```

```typescript
export function SafePaginatedEndpoint<T extends Type>(
  model: T,
  options: SafePaginatedEndpointOptions = {},
): MethodDecorator {
  const paginatedOpts: PaginatedOptions = {};
  if (options.maxLimit !== undefined) paginatedOpts.maxLimit = options.maxLimit;
  if (options.links !== undefined) paginatedOpts.links = options.links;

  const decorators: MethodDecorator[] = [
    ApiPaginatedSafeResponse(model, { description: options.description }),
    Paginated(paginatedOpts),
  ];

  if (options.sort) decorators.push(SortMeta());
  if (options.filter) decorators.push(FilterMeta());
  if (options.message) decorators.push(ResponseMessage(options.message));
  if (options.code) decorators.push(SuccessCode(options.code));
  if (options.errors?.length) decorators.push(ApiSafeErrorResponses(options.errors));
  if (options.deprecated) decorators.push(Deprecated(options.deprecated));

  return applyDecorators(...decorators);
}
```

## 3. Error Code Mapping Improvements

### Problem

1. Extending `DEFAULT_ERROR_CODE_MAP` requires writing a function instead of a simple object merge.
2. `errorCodeMapper` receives only `unknown`, forcing `instanceof` checks for status code.
3. `DEFAULT_ERROR_CODE_MAP` values are typed as `string` instead of string literals.

### 3.1 `errorCodes` Option

New declarative option on `SafeResponseModuleOptions`:

```typescript
export interface SafeResponseModuleOptions {
  // ... existing options ...

  /** Declarative error code map. Merged on top of DEFAULT_ERROR_CODE_MAP.
   *  Use for simple status-to-code mappings. For dynamic logic, use errorCodeMapper. */
  errorCodes?: Record<number, string>;
}
```

**Resolution order in the filter:**

1. `errorCodeMapper(exception, { statusCode, defaultCode })` — if returns string, use it
2. `options.errorCodes[statusCode]` — declarative override
3. `DEFAULT_ERROR_CODE_MAP[statusCode]` — built-in default
4. `'INTERNAL_SERVER_ERROR'` — final fallback

Implementation in `SafeExceptionFilter.catch()`:

```typescript
// Step 1: Try custom mapper
let errorCode: string | undefined;
if (this.options.errorCodeMapper) {
  try {
    const defaultCode = this.resolveDefaultCode(statusCode);
    errorCode = this.options.errorCodeMapper(exception, { statusCode, defaultCode });
  } catch {
    // Fall back to default resolution
  }
}

// Step 2: Declarative map
if (!errorCode && this.options.errorCodes) {
  errorCode = this.options.errorCodes[statusCode];
}

// Step 3: Built-in default
if (!errorCode) {
  errorCode = DEFAULT_ERROR_CODE_MAP[statusCode] ?? 'INTERNAL_SERVER_ERROR';
}
```

### 3.2 `errorCodeMapper` Signature Enhancement

**Current:**

```typescript
errorCodeMapper?: (exception: unknown) => string | undefined;
```

**New (backward-compatible):**

```typescript
export interface ErrorCodeMapperContext {
  /** Resolved HTTP status code */
  statusCode: number;
  /** Default code from errorCodes option or DEFAULT_ERROR_CODE_MAP */
  defaultCode: string;
}

errorCodeMapper?: (
  exception: unknown,
  context?: ErrorCodeMapperContext,
) => string | undefined;
```

The second argument is optional, so existing mappers `(exception) => ...` continue to work without changes.

### 3.3 Const Assertions

```typescript
// Before
export const DEFAULT_ERROR_CODE_MAP: Record<number, string> = { ... };
export const DEFAULT_PROBLEM_TITLE_MAP: Record<number, string> = { ... };

// After
export const DEFAULT_ERROR_CODE_MAP = {
  400: 'BAD_REQUEST',
  401: 'UNAUTHORIZED',
  403: 'FORBIDDEN',
  404: 'NOT_FOUND',
  405: 'METHOD_NOT_ALLOWED',
  409: 'CONFLICT',
  422: 'UNPROCESSABLE_ENTITY',
  429: 'TOO_MANY_REQUESTS',
  500: 'INTERNAL_SERVER_ERROR',
  502: 'BAD_GATEWAY',
  503: 'SERVICE_UNAVAILABLE',
} as const;

export const DEFAULT_PROBLEM_TITLE_MAP = {
  400: 'Bad Request',
  401: 'Unauthorized',
  // ...
} as const;
```

**Type compatibility note:** `as const` makes values readonly, but since these maps are only read (never mutated), this is safe. The maps are still indexable with `number` keys. Any code doing `DEFAULT_ERROR_CODE_MAP[statusCode]` will now get `string | undefined` instead of `string` (because arbitrary number keys may not exist) — but this is already handled with the `?? 'INTERNAL_SERVER_ERROR'` fallback everywhere.

### Private Helper

```typescript
private resolveDefaultCode(statusCode: number): string {
  return this.options.errorCodes?.[statusCode]
    ?? DEFAULT_ERROR_CODE_MAP[statusCode]
    ?? 'INTERNAL_SERVER_ERROR';
}
```

### Breaking Changes

### Migration Guide

No migration needed. Users can adopt new features incrementally:

```typescript
// Opt-in to composite decorators (replace manual stacking)
// Before
@ApiPaginatedSafeResponse(UserDto)
@Paginated({ maxLimit: 100 })
@ApiSafeErrorResponses([400, 404])
// After
@SafePaginatedEndpoint(UserDto, { maxLimit: 100, errors: [400, 404] })

// Opt-in to declarative error codes
// Before
errorCodeMapper: (ex) => ex instanceof HttpException && ex.getStatus() === 408 ? 'REQUEST_TIMEOUT' : undefined
// After
errorCodes: { 408: 'REQUEST_TIMEOUT' }
```
