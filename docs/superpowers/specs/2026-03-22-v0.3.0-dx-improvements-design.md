# v0.3.0 설계 문서 — DX 개선

## 개요

nestjs-safe-response v0.3.0은 세 가지 DX 개선 기능을 추가합니다:

1. **`@Exclude()` 공존 보장** — class-transformer의 ClassSerializerInterceptor와 자연스러운 공존
2. **커스텀 성공 코드 매핑** — 에러 코드 매핑의 대칭으로 성공 응답에도 의미론적 코드 부여
3. **응답 변환 훅** — 래핑 전 데이터 변환/필터링을 위한 `transformResponse` 옵션

모든 변경은 optional 필드 추가이므로 기존 사용자의 코드와 100% 하위호환됩니다.

---

## 1. 인터페이스 확장

### SafeResponseModuleOptions 변경

```typescript
export interface SafeResponseModuleOptions {
  // 기존 (변경 없음)
  timestamp?: boolean;
  path?: boolean;
  errorCodeMapper?: (exception: unknown) => string | undefined;
  dateFormatter?: () => string;

  // v0.3.0 신규
  successCodeMapper?: (statusCode: number) => string | undefined;
  transformResponse?: (data: unknown) => unknown;
}
```

| 필드 | 타입 | 설명 |
|------|------|------|
| `successCodeMapper` | `(statusCode: number) => string \| undefined` | HTTP 상태코드를 의미론적 성공 코드 문자열로 매핑 |
| `transformResponse` | `(data: unknown) => unknown` | 래핑 전 원본 데이터를 변환/필터링하는 훅 |

### SafeSuccessResponse 변경

```typescript
export interface SafeSuccessResponse<T = unknown> {
  success: true;
  statusCode: number;
  code?: string;      // v0.3.0 신규
  data: T;
  meta?: ResponseMeta;
  timestamp?: string;
  path?: string;
}
```

### 신규 상수

```typescript
// constants.ts
export const SUCCESS_CODE_KEY = 'SUCCESS_CODE';
```

### 신규 데코레이터

```typescript
// decorators/index.ts — SUCCESS_CODE_KEY를 ../constants에서 import 추가 필요
import { SUCCESS_CODE_KEY } from '../constants';

export const SuccessCode = (code: string) => SetMetadata(SUCCESS_CODE_KEY, code);
```

**`@SuccessCode()`는 메서드 레벨 전용 데코레이터입니다.** 기존 `@RawResponse()`, `@Paginated()`, `@ResponseMessage()`가 모두 `reflector.get(..., context.getHandler())`로 메서드 레벨만 읽는 패턴과 동일합니다. 클래스 레벨에 적용하면 무시됩니다. README에 메서드 데코레이터임을 명시합니다.

## 2. 인터셉터 로직 변경

### 실행 파이프라인 (변경 후)

```
컨트롤러 반환값 (data)
  → [신규] transformResponse(data)       // 래핑 전 변환
  → [기존] 래핑 { success, statusCode, data }
  → [신규] 성공 코드 해석                 // @SuccessCode > successCodeMapper
  → [기존] 페이지네이션 처리
  → [기존] 메시지 추가
  → [기존] 타임스탬프/경로 추가
  → 최종 응답 반환
```

### transformResponse 삽입 위치

`map()` 콜백 진입 직후, 래핑 전에 실행됩니다.

```typescript
map((data) => {
  if (this.options.transformResponse) {
    data = this.options.transformResponse(data);
  }
  // ... 기존 래핑 로직
})
```

페이지네이션 체크(`isPaginatedResult`)보다 앞이어야 합니다. `transformResponse`가 데이터 구조를 변경할 수 있으므로, 변경된 결과로 페이지네이션을 판단해야 합니다.

**엣지 케이스**: `transformResponse`가 `null` 또는 `undefined`를 반환하면, 해당 값이 그대로 `data`로 사용됩니다. 페이지네이션 체크에서 `isPaginatedResult(null)` → `false`가 되므로 일반 래핑으로 처리됩니다.

**에러 처리**: `transformResponse`가 예외를 던지면, 예외가 그대로 전파되어 `SafeExceptionFilter`에 의해 표준 에러 응답으로 변환됩니다. 별도의 catch/fallback 로직은 추가하지 않습니다.

**비동기 미지원**: `transformResponse`는 동기 함수만 지원합니다 (`(data: unknown) => unknown`). 만약 사용자가 async 함수를 전달하면 Promise 객체 자체가 `data`로 래핑됩니다. 비동기 변환이 필요한 경우 별도의 NestJS 인터셉터를 사용해야 합니다.

### 성공 코드 해석 로직

```typescript
// intercept() 상단에서 메타데이터 읽기
const successCode = this.reflector.get<string>(
  SUCCESS_CODE_KEY,
  context.getHandler(),
);

// map() 내부에서 code를 해석하고, 응답 객체 리터럴에 포함
let code: string | undefined = successCode;  // 데코레이터 우선
if (!code && this.options.successCodeMapper) {
  code = this.options.successCodeMapper(statusCode);
}

// code는 응답 객체 리터럴에 직접 포함하여 statusCode 바로 뒤에 위치
const response: SafeSuccessResponse = {
  success: true,
  statusCode,
  ...(code && { code }),  // statusCode 바로 뒤에 위치
  data,
};
```

**우선순위**: `@SuccessCode()` 데코레이터 > `successCodeMapper` 모듈 옵션 > 코드 없음(생략).
에러 쪽의 `errorCodeMapper` > `DEFAULT_ERROR_CODE_MAP` 우선순위와 동일한 패턴입니다.

## 3. @Exclude() 공존 보장

코드 변경 없이 **검증과 문서**로 해결합니다.

### NestJS 인터셉터 실행 순서

```
APP_INTERCEPTOR 등록 순서: [SafeResponseInterceptor, ClassSerializerInterceptor]
                            (SafeResponseModule 먼저 import)

요청 흐름 (onion model):
  요청 → SafeResponse.intercept() → ClassSerializer.intercept() → 컨트롤러
  응답 ← SafeResponse의 map()     ← ClassSerializer의 map()     ← 컨트롤러
```

ClassSerializerInterceptor의 `map()`이 먼저 실행되어 `@Exclude()` 필드를 제거하고,
SafeResponseInterceptor의 `map()`이 제거된 데이터를 래핑합니다.

### 필요 작업

1. E2E 테스트 (정상): `@Exclude()` 필드가 있는 DTO + ClassSerializerInterceptor 등록 → 응답에서 해당 필드가 제거되었는지 검증
2. E2E 테스트 (역순 경고 근거): ClassSerializer를 SafeResponse보다 먼저 등록 시 `@Exclude()` 필드가 남아있는지 확인. 이 테스트는 문서에서 등록 순서 경고의 근거로 사용
3. README 가이드: 모듈 등록 순서 안내, 역순 등록 시 주의사항 경고, 예제 코드

### 테스트용 DTO 예시

```typescript
import { Exclude } from 'class-transformer';

class UserResponse {
  id: number;
  name: string;
  @Exclude() password: string;
}
```

- 정상 순서 테스트: `res.body.data.password`가 `undefined`임을 확인
- 역순 테스트: `res.body.data.password`가 존재함을 확인 (문서 경고 근거)

## 4. Swagger DTO 변경

기존 `SafeSuccessResponseDto`에 `code` 프로퍼티만 추가합니다. 기존 필드(`success`, `statusCode`, `meta?`, `timestamp?`, `path?`)는 변경하지 않습니다.

```typescript
// dto/response.dto.ts — 기존 SafeSuccessResponseDto에 추가할 필드만 표시
@ApiPropertyOptional({ example: 'OK', description: 'Success code' })
code?: string;              // v0.3.0 신규 — statusCode 프로퍼티 아래에 삽입
```

## 5. 테스트 전략

### 유닛 테스트 (interceptor.spec.ts 추가)

| 테스트 케이스 | 설명 |
|--------------|------|
| `transformResponse` 적용 | 변환 함수가 data에 적용되는지 확인 |
| `transformResponse` 미설정 | 옵션 없으면 data가 그대로 전달되는지 확인 |
| `transformResponse` + 페이지네이션 | 변환 후 결과로 페이지네이션 판단 |
| `@SuccessCode()` 데코레이터 | 응답에 code 필드 포함 확인 |
| `successCodeMapper` 전역 매퍼 | statusCode → code 매핑 확인 |
| `@SuccessCode()` 우선순위 | 데코레이터가 전역 매퍼보다 우선 |
| 성공 코드 미설정 | 둘 다 없으면 code 필드 생략 확인 |
| `transformResponse` 예외 전파 | 변환 함수가 예외를 던지면 그대로 전파되어 에러 응답으로 변환됨 |
| `transformResponse` null 반환 | null 반환 시 data가 null로 래핑됨 (페이지네이션 미적용) |

### E2E 테스트

| 테스트 케이스 | 설명 |
|--------------|------|
| transformResponse E2E | HTTP 요청에서 변환 적용 확인 |
| successCodeMapper E2E | 응답에 code 필드 포함 확인 |
| @SuccessCode() E2E | 라우트별 코드 오버라이드 확인 |
| @Exclude() 공존 E2E (정상) | ClassSerializerInterceptor와 함께 사용 시 필드 제거 확인 |
| @Exclude() 역순 등록 E2E | ClassSerializer를 SafeResponse보다 먼저 등록하면 필드가 남아있는지 확인 |

- `test/e2e/test-app.module.ts`에 테스트 모듈 설정 추가 (transformResponse, successCodeMapper, ClassSerializer)
