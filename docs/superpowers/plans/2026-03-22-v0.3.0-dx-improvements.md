# v0.3.0 DX 개선 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** nestjs-safe-response에 응답 변환 훅, 커스텀 성공 코드 매핑, @Exclude() 공존 검증을 추가하여 DX를 개선한다.

**Architecture:** 기존 `SafeResponseModuleOptions` 인터페이스를 확장하고, 인터셉터의 `map()` 파이프라인에 두 가지 훅을 삽입한다. `@Exclude()` 공존은 코드 변경 없이 E2E 테스트와 문서로 보장한다. TDD로 진행하며, 각 기능을 독립적으로 구현/테스트/커밋한다.

**Tech Stack:** NestJS 11, TypeScript 5.7, Jest 30, supertest, class-transformer (devDependency for @Exclude() tests)

**Spec:** `docs/superpowers/specs/2026-03-22-v0.3.0-dx-improvements-design.md`

- [ ] **Step 3: 테스트 모듈 2개 추가**

`test/e2e/test-app.module.ts`에 import 추가:

```typescript
import { ClassSerializerInterceptor } from '@nestjs/common';
import { APP_INTERCEPTOR } from '@nestjs/core';
```

모듈 2개 추가:

NestJS에서 `APP_INTERCEPTOR`의 실행 순서는 DI 컨테이너에 등록되는 순서에 의해 결정된다. imported 모듈의 providers가 먼저 등록되고, 현재 모듈의 providers가 나중에 등록된다. 나중에 등록된 인터셉터의 `map()`이 먼저 실행된다 (onion model).

```typescript
// 정상 순서: SafeResponseModule(import)의 인터셉터가 먼저 DI에 등록되고,
// ClassSerializer(local provider)가 나중에 등록됨.
// → ClassSerializer.map()이 먼저 실행 → @Exclude() 적용 후 → SafeResponse.map() 래핑
@Module({
  imports: [SafeResponseModule.register()],
  controllers: [TestController],
  providers: [
    {
      provide: APP_INTERCEPTOR,
      useClass: ClassSerializerInterceptor,
    },
  ],
})
export class TestAppExcludeModule {}
```

역순 테스트를 위해서는 ClassSerializer를 별도 모듈로 분리하여 SafeResponseModule 보다 먼저 import해야 한다:

```typescript
// ClassSerializer를 별도 모듈로 분리
@Module({
  providers: [
    {
      provide: APP_INTERCEPTOR,
      useClass: ClassSerializerInterceptor,
    },
  ],
})
class ClassSerializerModule {}

// 역순: ClassSerializerModule이 먼저 import → 인터셉터가 DI에 먼저 등록
// → SafeResponse.map()이 먼저 실행 (래핑) → ClassSerializer.map() 실행 (래핑된 객체에 직렬화)
// → @Exclude()가 data 내부가 아닌 래핑된 최상위 객체에 적용되어 password가 살아남음
@Module({
  imports: [
    ClassSerializerModule,
    SafeResponseModule.register(),
  ],
  controllers: [TestController],
})
export class TestAppExcludeReversedModule {}
```

**주의**: 역순 테스트에서 `@Exclude()` 필드가 실제로 살아남는지 반드시 실행 후 확인한다. 만약 NestJS 버전에 따라 동작이 다르면 해당 테스트를 `.skip()` 처리하고 문서 경고만 유지한다.
